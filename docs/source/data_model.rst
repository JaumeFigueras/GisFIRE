Data Model
==========

.. contents::
   :local:
   :depth: 1

Overview
--------

The GisFIRE project needs a way to describe the entities involved in wildfire analysis â€”
lightning strikes, meteorological data, thunderstorms, wildfire ignitions â€” and to store
them in a spatial database. This is the **data model**.

The model is built on the general characteristics of each entity and the specifics of
the different data providers, so an object-oriented approach is used: generic domain
models with provider-specific subclasses.

What is a Data Model?
---------------------

A data model describes:

- The entities in the system (for example ``Lightning``, ``WeatherStation``,
  ``Thunderstorm``).
- The relationships between these entities.
- The properties and data types of each entity.
- Any constraints or business rules.

Here it is implemented in code with an ORM,
`SQLAlchemy <https://www.sqlalchemy.org>`_, and its spatial companion
`GeoAlchemy2 <https://geoalchemy-2.readthedocs.io>`_ on top of PostgreSQL/PostGIS.

Building blocks
---------------

Declarative base
^^^^^^^^^^^^^^^^^

Every ORM model inherits from a single :class:`~src.data_model.Base`, a plain
SQLAlchemy 2.0 :class:`~sqlalchemy.orm.DeclarativeBase` subclass (typed
:class:`~sqlalchemy.orm.Mapped` / :func:`~sqlalchemy.orm.mapped_column` columns).
Importing a model module registers its table on
:attr:`Base.metadata <sqlalchemy.orm.DeclarativeBase.metadata>`, from which the
schema is created directly
(:meth:`Base.metadata.create_all() <sqlalchemy.schema.MetaData.create_all>`) â€” there
are no hand-written SQL DDL files to keep in sync.

Primary keys
^^^^^^^^^^^^

Models declare their primary key as a bare
``id: Mapped[int] = mapped_column(primary_key=True)``, with no ``autoincrement``
argument. The column *is* autoincrementing all the same: SQLAlchemy defaults to
``autoincrement="auto"``, which applies to any single-column integer primary key that
has no other default, so on PostgreSQL the DDL comes out as ``id SERIAL NOT NULL``.
Passing ``autoincrement=True`` explicitly would change nothing. This is why models can
be inserted without ever setting ``id`` â€” the database assigns it and SQLAlchemy reads
it back through ``INSERT ... RETURNING``.

.. note::

   ``SERIAL`` is PostgreSQL's legacy mechanism. Since PostgreSQL 10 the standard
   equivalent is an identity column, written in SQLAlchemy as
   ``mapped_column(Identity(), primary_key=True)`` and emitted as
   ``id INTEGER GENERATED BY DEFAULT AS IDENTITY NOT NULL``. It behaves identically
   from the ORM's point of view, but the sequence is properly owned by the column (it
   is dropped with the table and needs no separate ``GRANT``), and an explicit
   ``INSERT`` of an ``id`` cannot silently desynchronise the sequence the way it can
   with ``SERIAL``. The project has not switched; if it ever does, the change is
   confined to the ``mapped_column`` calls and requires no test changes.

Providers and products
^^^^^^^^^^^^^^^^^^^^^^

A :class:`~src.data_model.data_provider.DataProvider` row does not stand for an
organisation but for one **data product** of that organisation. GWIS, for example,
publishes both the Global Wildfire Database and the Fine Fuel Moisture Code (FFMC);
tying ingested data to plain ``"GWIS"`` would be too coarse to tell the two apart.
Every row therefore carries a ``product`` alongside its ``name``, and it is the
``(name, product)`` pair that is unique â€” not ``name`` on its own.

The product is a plain string column rather than a separate table. The set of products
is small, they carry no attributes of their own, and a lookup table would add a join
and a second insert for no gain.

Generic and provider-specific models
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

Generic domain models (:class:`~src.data_model.data_provider.DataProvider`,
``Lightning``, ``Thunderstorm``,
``WeatherStation``, ``Variable``, ...) use polymorphic inheritance; provider-specific
subclasses add the columns particular to each data source.

API reference
-------------

Each model lives in its own module under ``src/data_model/`` and has its own reference
page, generated from the source with autodoc. New models get a page added to the list
below as they are ported.

:doc:`data_model/base`
    The declarative base class every ORM model inherits from. It collects the table
    definitions so the database schema can be created from the models.

:doc:`data_model/data_provider`
    A source of data ingested by GisFIRE (a weather service, a lightning-detection
    network, ...), narrowed down to one of its data products. Provider-specific data
    tables reference it, so data can only be attached to a known provider.

:doc:`data_model/wildfire`
    A wildfire event â€” when it started, when it was extinguished and the burnt area, kept
    as a PostGIS ``MULTIPOLYGON`` in EPSG:4326.
